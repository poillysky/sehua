# sehuatang 前后端架构

> 目标形态：**一个写库后端 + 两个前端**（管理端、搜索端）+ **PostgreSQL**，**全部部署在家庭 NAS**。  
> 管理前端 = **浏览器打开的网页**（可手机浏览），**不做 App**。  
> 配合 [设计说明.md](./设计说明.md)：采集自有、入库对齐 ed2k、不做 TG。  
> 部署细节见 [部署.md](./部署.md)。

---

## 1. 推荐总览

```text
                         家庭 NAS（Docker）
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   ┌──────────────────── PostgreSQL :5433 ─────────────────┐  │
│   │  ed2k_resources / resource_sources / crawl_* / auth_* │  │
│   └──────────────▲──────────────────────────▲─────────────┘  │
│                  │ 写                        │ 只读           │
│   ┌──────────────┴──────────┐    ┌───────────┴────────────┐  │
│   │ Backend（收集器）:8080   │    │ 搜索前端 next-web:3008 │  │
│   │ FastAPI + Playwright    │    │ Next.js + GraphQL+pg   │  │
│   │ 爬虫 / 解析 / 入库 / JWT│    │ 不经后端、不写库        │  │
│   └──────────────▲──────────┘    └────────────────────────┘  │
│                  │ REST /api/*                               │
│   ┌──────────────┴──────────┐                                │
│   │ 管理前端 admin :8081    │  （生产可经网关同域反代）         │
│   │ Nginx 静态 SPA          │                                │
│   └─────────────────────────┘                                │
└──────────────────────────────────────────────────────────────┘
         ▲ 局域网浏览器                    ▲（可选）爬虫出网代理
         │ http://NAS_IP:3008 / :8081      │ HTTP_PROXY / 旁路网关
```

### 与旧习惯的差异（已定）

| 旧习惯 | 现目标 |
|--------|--------|
| DB + 搜索在 NAS；爬虫 + 管理在 PC | **四件套全部在 NAS** |
| NAS 上 `docker compose build` | **CI 打镜像 → GHCR；NAS 只 `pull` + `up`** |
| 开发机常驻跑 Playwright | 开发机仅本地调试；生产爬虫跑在 NAS 容器 |

### 管理前端形态（已定）

| 做 | 不做 |
|----|------|
| **普通网页**，浏览器打开（PC / 手机） | **不做**原生 App、小程序、套壳 APK/IPA |
| 响应式布局 | 不以 PWA / 应用商店为交付目标 |
| SPA 静态资源（Nginx） | 不依赖应用商店分发 |

### 为何这样拆

| 原则 | 说明 |
|------|------|
| **PG 唯一真相** | 收集器写、搜索只读；搜索不必等 Admin API |
| **双前端职责正交** | 搜索 = 逛/搜/看；管理 = 爬/导/配/清 |
| **管理走 REST** | 爬虫启停、导入、重置、RBAC 适合 REST |
| **搜索走 GraphQL+pg** | 沿用 Next 方案；不强迫改走后端 |
| **NAS 全栈** | 单一运维面；局域网一把梭 |
| **无 TG** | 后端与两端 UI 均不出现群组监听 |

## 2. 仓库布局（Monorepo）

```text
sehuatang/
├── backend/                 # FastAPI 收集器（写路径 + Admin API）
├── frontend/
│   └── admin/               # 管理前端（Vite+React → 静态）
├── next-web/                # 搜索前端（Next.js :3008）
├── database/migrations/     # PG 迁移 001–017
├── deploy/                  # NAS 一键 compose（只 pull 镜像）
├── docs/
│   ├── 设计说明.md
│   ├── 架构.md              # 本文
│   └── 部署.md              # NAS / GHCR / 一键更新
└── start.bat                # 本机开发三窗启动（非生产）
```

| 服务 | 默认端口 | 角色 | 镜像来源 |
|------|----------|------|----------|
| PostgreSQL | 5433→5432 | 数据 | `postgres:16-alpine`（官方） |
| Backend | 8080 | 写库 + `/api/*` + Playwright | `ghcr.io/<org>/sehuatang-backend` |
| Admin | 8081→80 | 管理 SPA | `ghcr.io/<org>/sehuatang-admin` |
| Search | 3008→3000 | 搜索站 | `ghcr.io/<org>/sehuatang-search` |

`ed2k/` 目录视为**参考/历史镜像源**，生产以顶层 `backend` + `next-web` + `frontend/admin` 为准。

## 3. 后端（Backend）

### 3.1 职责

| 做 | 不做 |
|----|------|
| 站点登录 / 过验证 / 板块 / 列表 / 解析（自有） | 搜索页 SSR / SEO |
| `upsert_resource` 等对齐 ed2k 入库 | 给搜索提供主读库 GraphQL |
| Admin REST：资源、爬虫、导入、设置、系统 | Telegram |
| JWT + RBAC | 替代搜索端鉴权 |

### 3.2 模块

```text
backend/
  api/                 # FastAPI、CORS、静态预览
  auth/                # JWT（管理端专用）
  crawler/ + parsers/  # 自有采集与解析
  db/                  # repository / settings / crawl_*
  workers/             # 调度、pipeline、persist
```

路由分组（除 login/health 外均鉴权）：

| 前缀 | 用途 |
|------|------|
| `/api/auth/*` | 登录、用户、角色 |
| `/api/resources/*` | 处理记录 |
| `/api/crawler/*` | 状态、启停、循环、队列重试 |
| `/api/forum/*` | 板块与论坛配置、链路/解析探测 |
| `/api/import*` | 人工导入 |
| `/api/settings`、`/api/system/*` | 设置、概览、重置 |
| `/parse/thread` | 解析测试 |
| `/health` | 探活 |
| `/static/previews/*` | 预览图 |

对搜索前端：**零强依赖**。设置里可存 `search_frontend_url` 供管理台跳转。

### 3.3 NAS 上的特殊点

- 镜像含 **Chromium + Playwright + unrar**，体积大、内存敏感。
- 容器需 **`shm_size ≥ 2g`**，否则无头浏览器易崩。
- 论坛在境外或需翻墙时：配置 **HTTP 代理**（环境变量或设置页），代理跑在局域网其他机器即可。
- 数据卷持久化：`cookies`、预览上传、队列状态等 → NAS 共享目录。

### 3.4 CORS / 同域

| 环境 | 策略 |
|------|------|
| 本机开发 | Admin `:8081` Vite 代理 → Backend `:8080` |
| NAS 简单模式 | 分端口访问；Backend CORS 放行 `http://NAS_IP:8081` |
| **NAS 推荐** | 网关同域反代：`/admin` → admin，`/api` `/static` `/health` → backend；Cookie 同源 |

## 4. 搜索前端 `next-web`

| 项 | 约定 |
|----|------|
| 技术 | Next.js + Apollo GraphQL + `pg` 直连 |
| 数据 | **只读** `ed2k_resources` / `resource_sources` |
| 与 Backend | **不调用**爬虫/导入/重置 |
| 与管理端 | 无代码耦合；共用同一库 |
| 容器 | standalone 输出；`POSTGRES_*` / `POSTGRES_DB_URL` 指向 compose 内 `postgres` |

---

## 5. 管理前端 `frontend/admin`

### 5.1 定位

- 操作员 Web 后台：爬虫、导入、配置、处理记录、清数据。
- 只打 Backend REST，**不直连 Postgres**，不复用搜索 GraphQL。
- 生产镜像：`vite build` → **Nginx 托管静态文件**；`/api` 等由网关或 Nginx 反代到 backend。

### 5.2 路由（业务页）

`/login`、`/resources`、`/crawler`、`/parse-test`、`/data`、`/settings`

### 5.3 契约

- 列表字段与 `/api/resources/recent` 对齐。  
- **禁止**管理前端写 SQL；**禁止** TG 配置。

## 6. 数据流

```text
论坛 ──(可经代理)──► Backend 采集 ──► DualParseResult
                          │
                          ▼ persist
                   PostgreSQL（NAS 卷）
                     │         │
                     │         └──只读──► next-web GraphQL
                     │
                     └──REST──► admin（查看 / 控制 / 导入 / 设置）
```

人工导入：admin → `POST /api/import*` → 同一套 `upsert_resource`。

## 7. 鉴权

| 端 | 鉴权 |
|----|------|
| admin ↔ Backend | JWT（Cookie 或 Bearer）+ RBAC（admin / operator / viewer） |
| 搜索前端 | 默认可局域网公开只读；加密站以后单做，**不**复用操作员 Cookie |
| Backend ↔ 论坛 | Cookie / 会员会话，与管理 JWT 分离 |

## 8. 部署架构（NAS 全栈）摘要

详细步骤、Compose、内存与更新命令见 **[部署.md](./部署.md)**。

```text
开发机 / CI                         家庭 NAS
─────────                           ────────
git push ──► GitHub
              Actions build
              push GHCR  ──────►  docker login ghcr.io
                                  compose pull
                                  compose up -d
                                  （不在 NAS 上 build 业务镜像）
```

| 决策 | 选择 | 原因 |
|------|------|------|
| 部署面 | 全部 NAS | 一机运维、局域网访问 |
| 镜像构建地 | GitHub Actions → GHCR | 避免 NAS 编 Playwright（慢、易失败） |
| NAS 动作 | 只 pull / up | 「一键更新」脚本即可 |
| DB | 同 compose 内 postgres | 内网 DNS `postgres:5432`；宿主机仍可映射 5433 |
| 爬虫出网 | 可选 HTTP 代理 | 家庭宽带直连论坛常不稳定 |

### NAS 资源建议（起步）

| 项 | 建议 |
|----|------|
| CPU | x86_64 优先（Playwright 最省心）；ARM 可跑但更吃内存 |
| 内存 | **≥ 8GB** 给 Docker（后端 Chromium 峰值为主要占用） |
| 磁盘 | SSD/高速卷放 PG；预留预览图与日志 |
| shm | backend 容器 **≥ 2GB** |

## 9. 当前阶段建议

| 优先级 | 事项 |
|--------|------|
| P0 | 私有 GitHub 仓库首次 push；Actions 出三镜像 |
| P0 | NAS：`deploy/.env` + `update.sh` 拉起四服务 |
| P1 | 爬虫代理、改管理员密码、卷备份 |
| P1 | （可选）Caddy/Nginx 同域反代管理端 |
| P2 | 精简或归档 `ed2k/` 旧参考路径 |

**刻意不做：** 为管理去改搜索 GraphQL；把爬虫塞进 Next；上 TG；在 NAS 上日常 `compose build` 业务镜像。

## 10. 对比「不推荐」的形态

| 形态 | 原因 |
|------|------|
| 单前端既搜又管 | 权限与发布节奏缠死 |
| 搜索也经 Backend 读库 | 加重收集器、违背 Next+pg |
| 管理前端直连 PG | 写权限与 RBAC 风险 |
| NAS 本地 build Playwright | 耗时长、依赖网络差、难复现 |
| 爬虫常驻强依赖开发 PC | 与「全家 NAS」目标矛盾 |
