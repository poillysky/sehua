# sehuatang 设计说明

> 本文档固化当前产品边界：全面参考 `e:\ed2k` 收集器设计；站点采集链路自研；库表与写库语义对齐 ed2k；管理台功能分区对齐 ed2k 并适配手机；**不包含** Telegram / TG 群组能力。

**对照项目：** `ed2k`（历史参考：`backend/collector` + `database/migrations`）  
**本项目根目录：** `sehuatang/backend` + `frontend/admin` + `next-web`  
**前后端拆分：** 见 [架构.md](./架构.md)（Backend + 管理前端 + 搜索前端）  
**生产部署：** **家庭 NAS 全栈**（PG + 后端 + 管理 + 搜索）；镜像走 GHCR，详见 [部署.md](./部署.md)

---

## 1. 目标与原则

| 原则 | 说明 |
|------|------|
| 参考 ed2k | 管理 API、鉴权、资源表、写库语义、Admin 功能分区以 ed2k 为准 |
| 采集自有 | 登录、过验证、板块、列表、帖内解析用本项目实现 |
| 入库一致 | 解析结果写入 `ed2k_resources` + `resource_sources`（不用 `content_items`） |
| 不做 TG | 不实现、不迁移、不展示 Telegram 监听与群组配置 |
| 管理台可开手机 | **网页**（浏览器打开），分区与 ed2k 一致，响应式适配手机；**不做 App** |

## 2. 职责切分

### 2.1 自有（本项目实现 / 保留）

| # | 能力 | 现状入口（参考） | 说明 |
|---|------|------------------|------|
| 1 | 网站登录 / Cookie 会话 | `crawler/session.py`、`data/cookies.json` | 含浏览器引导拿 Cookie；若需论坛**会员登录**（隐藏帖/附件）在此扩展，与管理台 JWT 无关 |
| 2 | 过验证 | `crawler/fetcher.py` + session | CF、年龄门、R18 `safeid` → `_safe` 等站点门禁 |
| 3 | 板块读取 | `crawler/forum_index.py`、`parsers/boards.py` | 发现 fid / 名称；白名单与主链偏好（magnet / ed2k） |
| 4 | 列表读取 | `crawler/parser.py`（`parse_forum_list`） | 列表页扫帖、置顶过滤；产出待抓 `tid` / URL |
| 5 | 页面内容解析 | `parsers/*`（`parse_thread_dual`） | 标题、标签描述、密码、预览图、magnet + ed2k 双链、主链选择 |

### 2.2 对齐 ed2k（复用语义 / 迁移表结构）

| 能力 | ed2k 参考 | 本项目要求 |
|------|-----------|------------|
| 数据库迁移 | `e:\ed2k\database\migrations`（001–013） | 直接复用或拷贝到本仓库；不另起 `content_items` / `assets` 为主模型 |
| 写库入口 | `db/repository.py`：`upsert_resource`、`import_thread_resource`、`import_thread_stub` | 自有解析结果经**适配层**写入相同字段语义 |
| 爬取队列状态 | `crawl_pages` / `crawl_boards`、`link_kind` | 列表入队、详情完成后标记状态，与管理台「爬虫状态」一致 |
| 鉴权 | `auth/*`（JWT Cookie + RBAC） | 挂到 FastAPI；保护 Admin 与 `/api/*` |
| 管理 API | `workers/app.py` 中资源 / 爬虫 / 设置 / 导入 / 系统等 | 同构接口，**去掉全部 TG 相关字段与路由行为** |
| 管理静态页 | `static/admin` | 功能分区对齐；删除群组管理；布局适配手机 |

### 2.3 架构示意

```text
┌─ 自有采集 ─────────────────────────────────────────┐
│  登录/Cookie │ 过验证 │ 板块 │ 列表 │ 帖内解析       │
│              → DualParseResult（结构化）              │
└──────────────────────────┬──────────────────────────┘
                           │ 适配层 persist（对齐 ed2k）
                           ▼
┌─ 对齐 ed2k（无 TG） ────────────────────────────────┐
│  ed2k_resources + resource_sources + crawl_*         │
│  FastAPI：auth / resources / crawl / settings / …    │
│  Admin：处理记录 / 爬虫 / 解析测试 / 数据 / 设置      │
└─────────────────────────────────────────────────────┘
```

## 3. 解析 → 入库（与 ed2k 一致）

### 3.1 核心约定

- **一帖成功** ≈ 入库 **1 条主资源**（`ed2k_resources`）+ **1 条来源**（`resource_sources`）。
- magnet / ed2k **共用同一套表**；种类由主链 URI 前缀推断（`magnet:` / `ed2k://` / `unavailable://thread/`）。
- `source` 仅使用 **`web:crawler`**（爬虫）与 **`upload:manual`**（人工导入）。**不使用** `telegram:listener`。
- **废弃**本项目现有面向 `content_items` + `assets` 的持久化路径；管理台只认 hash 中心模型。

### 3.2 推荐流水线

```text
HTML（列表/详情已读到）
  → parse_thread_dual(html, preferred_link=板块策略)
  → DualParseResult
  → 适配层 persist_dual_parse(...)
        有主链 → upsert_resource（主链 + 全量同类链接数组）
        无主链但需留痕 → import_thread_stub 语义
  → 更新 crawl_pages（status / outcome / link_kind）
```

说明：适配层应使用**已解析结果**直接组装写入参数，避免再把整页正文丢回 ed2k `parse_text` 二次解析，防止两套正则结果不一致。ed2k 的 `import_thread_resource(text=…)` 仅作语义参考。

### 3.3 `DualParseResult` → 表字段映射

| DualParseResult / 上下文 | 写入目标 |
|--------------------------|----------|
| 主链 `hash` / `filename` / `size` / `uri` | `ed2k_resources.hash` / `filename` / `size` / `ed2k_link` |
| 同类全部 URI（主链为 magnet 则全部 magnets；为 ed2k 则全部 ed2k） | `resource_sources.ed2k_links[]` |
| `title` | `resource_sources.title` |
| `description` | `resource_sources.description` |
| `preview_images` | `resource_sources.preview_images` |
| `extract_password` | `resource_sources.extract_password` |
| 帖子 URL | `resource_sources.source_url` |
| `board_fid` / 板块名 | `resource_sources.board_fid` / `board_name` |
| `"sehuatang"` | `resource_sources.forum_id` |
| `filename + title + description + password` | `ed2k_resources.search_string`（与 ed2k `build_search_string` 一致） |

### 3.4 主链选择（与 ed2k `prefer_link_kind` 一致）

| 板块策略 | 行为 |
|----------|------|
| 偏好 `ed2k` | 有 ed2k → 主资源为 ed2k，`ed2k_links` 放全部 ed2k；无 ed2k → 不按 magnet 凑数（返回 0 或走 stub 策略） |
| 偏好 `magnet` | 有 magnet → 主资源为 magnet（`hash=infohash`），数组放全部 magnet |
| 默认 | **一帖一条**主资源（对齐 ed2k `web_crawler_one_link_per_thread`）；不默认「每条链拆多行主表」 |

### 3.5 Stub（无有效下载链）

与 ed2k `import_thread_stub` 一致：

| 字段 | 规则 |
|------|------|
| `hash` | 由帖 URL 派生（sha256 等，与 ed2k `thread_stub_hash` 一致） |
| `ed2k_link` | `unavailable://thread/{hash}` |
| `ed2k_links` | `[]` |
| 其它 | 仍可写 title / description / preview_images，便于「处理记录」可见 |

`crawl_pages.link_kind` 使用：`magnet` | `ed2k` | `stub` | `failed` | `skipped`（与 ed2k 一致）。

### 3.6 与界面的关系

| 场景 | 是否入库 |
|------|----------|
| 「解析测试」：提交 HTML / URL 看结构 | 默认**不入库**，返回 DualParseResult 风格 JSON |
| 爬虫详情完成 / 显式「确认入库」 | 走适配层写入 |
| 人工粘贴 / 文件导入 | `upload` 源，共用 `ed2k_resources` |

## 4. 管理台要求

### 4.1 功能分区（与 ed2k 对齐，去掉 TG）

| 分区 | 保留 | 说明 |
|------|------|------|
| 处理记录 | ✅ | 资源列表、筛选、详情、人工导入 |
| 爬虫状态 | ✅ | 启停、跑一轮、活动日志（仅论坛） |
| 解析测试 | ✅ | 单帖/HTML 解析调试 |
| 数据管理 | ✅ | 概览、重置 |
| 设置 · 账号 | ✅ | 登录用户 / RBAC |
| 设置 · 论坛 | ✅ | 板块、配置、拓扑、链路探测 |
| 设置 · 通用 | ✅ | 全局采集参数、代理探测等 |
| 设置 · **群组管理** | ❌ | 整块删除（TG） |
| 爬虫页 TG 占位 / 侧栏「TG 群组」 | ❌ | 不展示 |
| 来源筛选 `telegram` | ❌ | 仅保留论坛 `web`、人工 `upload` |

### 4.2 手机展示（网页，非 App）

- 交付形态：**浏览器打开的管理站点**，不做原生 App / 小程序 / 套壳安装包。
- 业务分区与上表一致，不因手机而砍功能模块。
- 布局从桌面双栏改为窄屏友好：顶栏/底栏导航、筛选抽屉、单列列表 + 全屏详情、设置纵向分段、宽表关键列优先。
- 实现时在现有 `static/admin` 或独立 `frontend/admin` SPA 上改 CSS/交互即可，仍是网页。

### 4.3 API 挂载前提

Admin 静态资源必须对应同构 `/api/*`（含 StaticFiles、JWT middleware）。当前仅有 `/health` + `/parse/thread` 时，管理页无法可用——需补齐资源/爬虫/设置/导入/系统接口后再改移动端样式。

## 5. 明确排除：Telegram / TG 群组

以下内容**不迁移、不实现、不展示**：

| 层 | 排除项 |
|----|--------|
| 进程 / 依赖 | Telethon、`tg_listener`、启动时 TG 线程、`requirements` 中的 telethon |
| 配置 | `TG_ENABLED`、`TG_API_ID`、`TG_API_HASH`、`TG_GROUPS`、`TG_SESSION` |
| API / Settings | `tg_enabled`、`tg_api_*`、`tg_groups`、`tg_session`；源类型 `telegram` |
| DB 源 | `telegram:listener`（无需 seed、无需使用） |
| Admin UI | 「群组管理」、爬虫 TG 占位、侧栏 TG 筛选、telegram 来源图标与文案 |

资源统一入口文案改为：**论坛爬取 + 人工导入 → PostgreSQL → 管理台**（无 TG）。

## 6. 与现状的差距（落地清单）

| 项 | 说明 |
|----|------|
| 模型 | 停止扩展 `content_items`；改为 ed2k 表 + 写库语义 |
| 迁移 | 引入 ed2k `database/migrations`（或共享同一 Postgres） |
| 适配层 | 新增 `DualParseResult` → `upsert_resource` / stub |
| Pipeline | 列表(自有) → 入队 → 详情解析(自有) → persist → 标记 `crawl_pages` |
| API | 挂载 auth + 管理用路由；爬虫执行调用本项目 runner |
| Admin | 去 TG；后补手机布局 |
| 依赖 | 补齐 JWT/bcrypt 等；**不**加 telethon |

## 7. 建议实施顺序

1. 接入 ed2k 迁移与 DSN，确认可写 `ed2k_resources`。  
2. 实现 `persist_dual_parse`（对齐 §3），替换 `content_items` 写路径。  
3. 贯通一条路径：自有抓取/解析 → 入库 → `/api/resources/recent` 可见。  
4. 挂载管理 API + Admin（去掉 TG）。  
5. 补全列表入队与常驻/按需爬虫循环。  
6. Admin 移动端布局优化。  
7. 按需增强：论坛会员登录、附件/rar、更细 stub 判定（均可仍落在「自有采集」侧）。

## 8. 术语

| 术语 | 含义 |
|------|------|
| DualParseResult | 本项目帖内双链解析结果（`parsers/links.py`） |
| 适配层 / persist | 将 DualParseResult 写成 ed2k 表结构的模块 |
| 主链 | 写入 `ed2k_resources` 的那一条 magnet 或 ed2k |
| Stub | 无下载链时的占位资源（`unavailable://thread/…`） |
| 管理台 JWT | `auth/*` 操作员登录，**不是**色花堂站点账号 |
